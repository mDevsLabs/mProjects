# Internationalization Implementation Guide

Welcome to the mAI Internationalization Implementation Guide. This document will guide you through understanding the internationalization mechanism of mAI, including file structure and how to add new languages. mAI uses `i18next` and `@lobehub/i18n-cli` as the internationalization solution, aiming to provide users with seamless multilingual support.

## Internationalization Overview

Internationalization (i18n for short) is the process of enabling an application to adapt to different languages and regions. In mAI, we support multiple languages and achieve dynamic language switching and content localization through the `i18next` library. Our goal is to provide a localized experience for global users.

## File Structure

In the mAI project, internationalization-related files are organized as follows:

- `packages/locales/src/default`: Contains translation files for the default development language (English), which we use as the source language (`entryLocale: 'en-US'` in `.i18nrc.js`).
- `locales`: Contains folders for all supported languages, with each language folder containing the respective translation files generated by `@lobehub/i18n-cli`.

In the directory structure of `packages/locales/src/default`, each file is a TypeScript namespace module containing the source (English) strings, while each language folder under `locales/` contains JSON translation files for the respective language. The files in each language folder correspond to the TypeScript namespace files in `packages/locales/src/default`, ensuring consistency in the structure of translation files across languages.

```bash
packages/locales/src/default
├── index.ts
├── agent.ts
├── chat.ts
├── common.ts
├── error.ts
├── setting.ts
├── tool.ts
├── welcome.ts
└── ... (50+ namespace files)
```

The file structure generated by `@lobehub/i18n-cli` is as follows:

```bash
locales
├── ar
│   ├── chat.json
│   ├── common.json
│   ├── error.json
│   └── ... (other translation files)
├── de-DE
│   ├── chat.json
│   ├── common.json
│   ├── error.json
│   └── ... (other translation files)
├── en-US
├── ... (other language directories)
├── zh-CN
└── zh-TW
```

## Core Implementation Logic

The internationalization core implementation logic of mAI is as follows:

- Initialize and configure using the `i18next` library.
- Automatically detect the user's language preference using `i18next-browser-languagedetector`.
- Dynamically load translation resources using `i18next-resources-to-backend`.
- Set the direction of the HTML document (LTR or RTL) based on the user's language preference.

Here is a simplified pseudo code example to illustrate the core implementation logic of internationalization in mAI:

```ts
import i18n from 'i18next';
import LanguageDetector from 'i18next-browser-languagedetector';
import resourcesToBackend from 'i18next-resources-to-backend';
import { isRtlLang } from 'rtl-detect';

// Create i18n instance and configure
const createI18nInstance = (lang) => {
  const i18nInstance = i18n
    .use(LanguageDetector) // Use language detection
    .use(
      resourcesToBackend((language, namespace) => {
        // Dynamically load translation resources for the corresponding language
        return import(`path/to/locales/${language}/${namespace}.json`);
      }),
    );

  // Listen for language change events and dynamically set document direction
  i18nInstance.on('languageChanged', (language) => {
    const direction = isRtlLang(language) ? 'rtl' : 'ltr';
    document.documentElement.dir = direction; // Set HTML document direction
  });

  // Initialize i18n instance
  i18nInstance.init({
    // Relevant configurations
  });

  return i18nInstance;
};
```

In this example, we demonstrate how to use `i18next` and related plugins to initialize internationalization settings. We dynamically import translation resources and respond to language change events to adjust the text direction of the page. This process provides mAI with flexible multilingual support capabilities.
